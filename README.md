# 💞 RelationSense · 关系感知

让 Bot 拥有自己的情感——对每一段关系，Bot 有自己的好感、信任、依恋。

**RelationSense** 是一个关系感知插件，让 Bot 在持续对话中形成自己的情感：  
**Bot 对这个人有多喜欢、多信任、多想他在身边，以及此刻该用什么样的心情回应。**

它不只是一个后台数据面板，而是一份会动态进化、从 Bot 视角注入每次对话的「情感记忆」。

---

## ✨ 核心理念：LLM 驱动的情感感知

RelationSense 的核心不是冰冷的数值和等级，而是**让对话模型自己感知关系**。

传统做法是后台模型算出数值 → 推导氛围 → 注入提示词，但推导出的氛围往往和对话模型的真实感受脱节。RelationSense 的 `<rs>` 标签机制让对话模型自己输出此刻的关系感知——氛围、语气、对方状态——插件解析后直接用于下次注入，实现**感知-注入闭环**。

后台模型仍然负责五维数值的量化追踪，但氛围和语气由对话模型亲自决定。

---

## 🧠 双模型分工

RelationSense 涉及两个不同的 LLM 角色，各司其职：

### 对话模型（用户配置的 Bot 模型）

就是 Bot 用来和用户聊天的模型——它**亲自扮演**角色、亲自感受关系。

**职责**：
- 正常回复用户消息
- 开启 `enable_live_perception` 后，在回复末尾输出 `<rs>` 标签，表达自己此刻的关系感知
- 开启 `enable_live_perception_update` 后，可通过 `<update>` 标签修正过时的感知

**它输出的内容**：
```
<rs>{"atmosphere": "亲密、依赖", "tone": "撒娇、黏人", "user_state": "正在和我闲聊"}</rs>
```
- `atmosphere`：Bot 感受到的关系氛围（如"亲密、依赖""暧昧、试探""冷淡、疏远"）
- `tone`：Bot 认为自己应该用的语气（如"撒娇、黏人""温柔、关心""冷淡、敷衍"）
- `user_state`：Bot 对对方当前状态的判断（如"正在和我闲聊""心情低落需要安慰"）

### 后台分析模型（`analysis_provider_id` 配置）

专门用于关系量化分析的独立模型——它**不参与对话**，只在后台定期评估五维数值变化。

**职责**：
- 定期分析对话记录，评估五维数值（好感度、信任度、对话深度、依赖度、回归率）的变化
- 当 `<rs>` 感知可用时，参考对话模型的关系评估来校准数值
- 输出结构化 JSON，包含每个维度的分数、变化原因、关系总结等

**它和对话模型的关系**：
- 对话模型说"我觉得关系很亲密" → 后台模型据此把好感度从 60 调到 72
- 对话模型说"对方心情低落" → 后台模型在 `user_state` 中记录，下次注入时使用
- 后台模型**不会推翻**对话模型的感知，只会用数值来量化表达

## 🏷️ `<rs>` 标签与 `<update>` 标签

### `<rs>` 标签 — 完整关系感知

`<rs>` 是对话模型输出的**完整关系感知**，包含三个字段：

```
<rs>{"atmosphere": "关系氛围", "tone": "回应语气", "user_state": "对方状态"}</rs>
```

**生命周期**：
1. 对话模型在回复中输出 `<rs>` 标签
2. 插件解析标签内容，存入数据库（`rs_content_{session_id}`）
3. 从回复文本中移除标签，用户不会看到
4. 下次注入时，`<rs>` 内容替代后台推导的氛围/语气/对方状态
5. 后台分析时，`<rs>` 内容作为参考传递给分析模型

**触发条件**：开启 `enable_live_perception` 后，对话模型每次回复都会被要求输出 `<rs>` 标签。

### `<update>` 标签 — 轻量感知修正

`<update>` 是对话模型输出的**轻量感知修正**，只更新对方状态：

```
<update>对方现在正在忙但还在回我</update>
```

**用途**：当对话模型发现之前感知的对方状态已经过时（比如几小时前感知到"在闲聊"，但对方现在明显在忙），可以用 `<update>` 快速修正，而不需要重新输出完整的 `<rs>` 标签。

**触发条件**：需要同时开启 `enable_live_perception` 和 `enable_live_perception_update`。

### `<rs>` 与 `<update>` 的优先级

- **同一轮回复中同时出现 `<rs>` 和 `<update>`**：只处理 `<rs>`，忽略 `<update>`（因为 `<rs>` 已经包含了完整的 user_state，`<update>` 是冗余的）
- **只出现 `<update>`**：只更新 `_live_user_state`（内存中的对方状态缓存），不更新数据库中的完整 `<rs>` 内容
- **只出现 `<rs>`**：更新数据库中的完整关系感知，同时更新 `_live_user_state`

## 🔄 数据流全景

```
用户发消息
    │
    ▼
┌─────────────────────────────────────────┐
│  on_llm_request（请求前）                  │
│  ├─ 缓存用户消息到 buffer                  │
│  ├─ 从 DB 读取关系状态                      │
│  ├─ 从 DB 读取 <rs> 内容（如有）            │
│  ├─ 构建注入文本（场景策略 or <rs> 驱动）    │
│  │   ├─ <rs> 可用 → 使用对话模型的氛围/语气  │
│  │   └─ <rs> 不可用 → 使用后台推导的氛围     │
│  └─ 注入到 system_prompt                   │
└─────────────────────────────────────────┘
    │
    ▼
对话模型生成回复（可能包含 <rs> / <update> 标签）
    │
    ▼
┌─────────────────────────────────────────┐
│  on_llm_response（回复后）                 │
│  ├─ 缓存 Bot 回复到 buffer                 │
│  ├─ 解析 <rs> 标签 → 存入 DB               │
│  ├─ 解析 <update> 标签 → 更新内存缓存       │
│  ├─ 从回复文本中移除标签                     │
│  └─ 检查是否触发后台分析                     │
└─────────────────────────────────────────┘
    │
    ▼（异步，不阻塞回复）
┌─────────────────────────────────────────┐
│  后台分析模型                               │
│  ├─ 读取 buffer 中的对话记录                │
│  ├─ 读取 <rs> 内容作为参考（如有）           │
│  ├─ 评估五维数值变化                        │
│  └─ 保存到 DB → 下次注入时使用              │
└─────────────────────────────────────────┘
```

## ⚙️ 三个感知配置项的关系

| 配置项 | 依赖 | 效果 |
|--------|------|------|
| `enable_live_perception` | 无 | 对话模型输出 `<rs>` 标签，插件解析后用于注入 |
| `enable_live_perception_update` | `enable_live_perception` | 额外允许对话模型输出 `<update>` 标签修正过时感知 |
| `enable_dialogue_static_update` | `enable_live_perception` | **始终**用对话模型的 `<rs>` 驱动注入内容，后台模型只管数值 |

### `enable_dialogue_static_update` 详解

这个开关决定了**注入内容的来源**：

**关闭时**（默认）：
- 初始化阶段：等待对话模型输出 `<rs>` → 用 `<rs>` 内容初始化注入 → 后台模型初始化数值
- 后续阶段：后台模型分析后输出 `user_state` / `tone_hint` → 注入时使用后台模型的结果
- 如果后台模型还没分析过，注入内容可能为空

**开启时**：
- 始终使用对话模型的 `<rs>` 内容来填充注入中的氛围、语气、对方状态
- 后台模型只负责五维数值的更新，不再影响注入文本
- 效果：Bot 的语气和氛围完全由对话模型自己决定，更连贯、更自然

## 📊 五维情感指标（带维度惯性）

RelationSense 用五个维度刻画 Bot 对用户的情感状态，且每轮分析变化受惯性限制，不会一夜之间从陌路跳到亲密：

| 维度 | 含义 | 单轮上限 |
|------|------|----------|
| **好感度** | Bot 对用户的喜欢程度 | ±12 |
| **信任度** | Bot 觉得这个人是否值得交心 | ±8 |
| **对话深度** | Bot 主观感受到的交流质量 | ±20 |
| **依赖度** | Bot 对用户的依恋——想不想他在 | ±10 |
| **回归率** | Bot 对「对方会回来」的期待感 | ±5 |

五维数值由后台模型量化追踪，但**氛围和语气由对话模型亲自感知**——这才是 Bot 真正"感受到"的东西。

---

## 🎯 情感等级

基于三核加权（好感度 ×0.4 + 信任度 ×0.35 + 对话深度 ×0.25）判定关系等级，作为后台参考指标：

| 等级 | 标签 | Bot 对你的真实感受 |
|------|------|-------------------|
| Lv-2 | 敌意 | Bot 对你怀有反感——因为持续被伤害 |
| Lv-1 | 厌恶 | Bot 对你感到厌烦——因为一直敷衍 |
| Lv0 | 冷淡 | Bot 对你没什么感觉——关系没进展 |
| Lv1 | 陌路 | Bot 觉得你是陌生人——礼貌但疏离 |
| Lv2 | 初识 | Bot 刚开始认识你——试探、好奇 |
| Lv3 | 认识 | Bot 觉得和你算认识了——友好、愿意聊 |
| Lv4 | 朋友 | Bot 把你当朋友——放松、主动关心 |
| Lv5 | 好友 | Bot 把你当好朋友——信任你、流露真实情感 |
| Lv6 | 亲密 | Bot 已经很喜欢你了——撒娇、依赖、也会宠 |

> 等级是量化参考，不是 Bot 的全部感受。对话模型的 `<rs>` 感知才是 Bot 此刻真实的情感状态。

## 📡 实时注入（多场景策略）

每次 LLM 请求时，RelationSense 会根据**当前场景**自动选择注入策略：

| 场景 | 判定条件 | 注入重点 |
|------|---------|---------|
| `<rs>` 驱动 | 对话模型输出了 `<rs>` 标签且含 atmosphere | 直接使用对话模型感知的氛围和语气 |
| 极简 | 好感 ≥90 且信任 ≥88 | "一切如常"，点到即止 |
| 冲突 | 好感度单轮下降 ≥ 5 | 别硬聊，留一句柔软信号即可 |
| 暧昧 | 好感 60-75，信任 < 70 | 带试探性的靠近，别追得太紧 |
| 沉寂 | 长时间不活跃 / 对方话少 | 别主动找话题，对方想说自然会说 |
| 正常 | 其他情况 | 常规 2-3 行关系上下文 |

> 当 `enable_dialogue_static_update` 开启且 `<rs>` 有 atmosphere 时，始终使用 `<rs>` 驱动模式，跳过场景判定。

## 🔍 智能触发 + 人格感知 + 手动分析

分析触发条件：
- **常规触发**：距上次 ≥ 30 分钟 + 新增 ≥ 15 条消息
- **回复触发**：Bot 即将回复时，若说话者超过 30 分钟未分析，异步触发
- **手动触发**：发送 `关系分析` 指令 → 立即分析

每次分析时**自动传入 Bot 人设**——温柔学妹对批评更敏感、高冷御姐好感来得更慢，分析结果贴合角色。

## 🔄 对话冷却

长时间没活跃？Bot 的热度也会自然下降——对话深度和依赖度每小时缓慢衰减。  
这不是冷暴力，是自我保护：你在忙你的生活，我对你的期待自然也调低了。

## 🌼 适配场景

如果你希望 Bot：
- 拥有自己的情感——好感、信任、依恋，不是回复机器
- **对话模型自己感知关系氛围**，而不是靠后台推导
- 在人设下做出符合性格的情感反应
- 不会一夜之间从陌路跳到亲密（惯性保护）
- 不同场景下自动切换语气（冲突≠暧昧≠沉寂）
- 在对方消失太久后，热度自然降温（冷却）

那 RelationSense 会很适合你。

## ⚙️ 主要配置项

### 基础开关
- `enable_injection`：是否注入关系上下文（默认开启）
- `debug_mode`：调试模式，日志中输出完整注入内容

### 对话模型感知（需 AstrBot v4.24.2+）
- `enable_live_perception`：开启对话模型自主感知（`<rs>` 标签机制）
- `enable_live_perception_update`：允许对话模型通过 `<update>` 标签修正过时感知（需先开启 `enable_live_perception`）
- `enable_dialogue_static_update`：对话模型全权驱动静态注入内容（需先开启 `enable_live_perception`）。开启后氛围/语气/对方状态始终由 `<rs>` 驱动，后台模型只管数值

### 分析模型
- `analysis_provider_id`（必填）：用于后台关系分析的 LLM 提供商
- `analysis_secondary_provider_id`：备用分析模型，主模型失败时自动尝试
- `analysis_timeout_seconds`：单次分析调用超时时间

### 触发参数
- `analysis_interval_minutes`：两次分析的最小间隔
- `min_new_messages`：新增消息数达到此值才可能触发分析

### 群聊模式
- `enable_group_mode`：开启后群聊自动切换多用户关系感知
- `unify_cross_session`：同一用户所有会话共享五维数据
- `group_active_days` / `group_analysis_interval_minutes` / `group_max_active_users`

### 存储
- `buffer_max_size`：每个会话在内存中缓存的最大消息条数

## 👥 群聊模式

开启 `enable_group_mode` 后，RelationSense 在群聊场景自动切换到多用户关系感知逻辑：

- **独立关系状态**：群内每个用户维护独立的五维数据和关系等级
- **群对话组装**：自动收集群内所有用户的消息（含 Bot 回复），组装完整群对话供分析
- **批量分析**：后台定期对过期活跃用户进行批量分析，单次 LLM 调用可分析多用户
- **回复触发**：Bot 即将回复时，若说话者超过 30 分钟未分析，自动触发异步分析
- **频率限制**：同一用户 30 分钟内不会重复分析，避免浪费 LLM 调用
- **活跃用户摘要**：注入时附带群内其他活跃用户的关系概要
- **`<rs>` 标签支持**：群聊场景同步支持对话模型的关系感知输出，含专用群聊 `<rs>` 驱动模板

> 私聊行为在群聊模式下完全不变。

## 📝 使用说明

1. 必须配置 `analysis_provider_id`，否则后台分析无法执行。
2. 分析异步后台运行，不阻塞主回复。
3. 同一会话串行（asyncio.Lock 防并发），`关系分析` 可在指令触发后立即查看进度。
4. 冷却衰减自动运行，无需干预。
5. `debug_mode` 开启后日志中会输出完整注入提示词内容。
6. `<rs>` 标签机制需要对话模型支持在回复中输出结构化标签，建议使用能力较强的模型。
7. `enable_dialogue_static_update` 开启后，如果对话模型长时间未输出 `<rs>` 标签，注入内容可能为空——确保对话模型能稳定输出 `<rs>`。
